20200311-稠密索引

稠密索引

思想

高考后，A考生对成绩存疑，请求复审。
官方人员接受请求后，是如何从庞大的考生数据库中提取出A考生的信息呢？

通常，考生信息至少应该包括如下信息：

  • 姓名
  • 性别
  • 学号
  • 学校
  • 准考证号
  • 数学成绩
  • 语文成绩
  • 英语成绩
  • 数学答题卷
  • 语文答题卷
  • 英语答题卷
  • … …

且当前考生数据库已经根据成绩高低，降序排序。

现在，A考生提供其准考证号，官方人员需要根据准考证号从考生数据库中检出A考生信息。虽然现在考生数据库已经有序，但要注意，其是根据成绩排序，而非根据准考证号排序，因此对于准考证号来说，考生数据库是无序的。

那么，官方人员可以在考生数据库中从头开始，逐条搜索并比对关键字（准考证号），这也就是前面提及的顺序查找。

而查找算法中，顺序查找的效率最为低下，而二分查找、插值查找、斐波那契查找虽然高效但要求数据已经按照关键字有序排列。

因此，官方人员打算按照关键字（准考证号），重新对数据库进行排序，但他又发现如下问题：

  • 考生数据库太大，无法全部加载到内存。（数据库过大）
  • 上司不允许要求现有的考生数据库必须是按照成绩，降序排序。（不允许改变原始数据库）
  • 最后，发现如果重新排序数据库，那么每排序一条记录都需要搬移较大的数据，那还不如顺序查找，反正每条数据都要处理。（数据库中的每条记录都很大，移动记录费时）

怎么办？官方人员请教运维人员，运维人员告知，考生数据库中的信息已经按照不同的关键字生成相应的索引表，且索引已经有序，只需要根据相应索引表进行二分查找即可。

官方人员奇了，什么是索引表？
首先索引就是把一个关键字与它对应的记录相关联的过程。具体的，就是创建一个数据结构，里面包括准考证号（关键字）和指向考生数据库中该考生信息（记录）的地址。
而索引表就是将若干索引按照关键字有序排列的结构。

官方人员问道，那是不是只要找到关键字为准考证号的索引表，再对该索引表进行二分查找，就可以找到A考生的索引，最后根据该索引中的地址去考生数据库中检出A考生信息就行。

是的，索引就是起这样的功能，而现在索引表用的是稠密索引方式。运维人员解释道。
所谓的稠密索引，是指在线性索引中，将数据集中的每个记录对应一个索引项。
且对于稠密索引的索引表来说，索引一定是按照关键码有序排列的。

20200310-斐波那契查找

斐波那契查找

思想

斐波那契查找是二分查找的改进算法，本质上，也是一种对有序数组进行分割查找的方法。
与二分查找不同的是，斐波那契查找是根据黄金分割比例分割有序数组。

什么是黄金分割比例？

黄金分割是指将整体一分为二，较大部分与整体部分的比值等于较小部分与较大部分的比值，其比值约为0.618。

如何根据黄金分割比例分割有序数组？

使用乘法计算

根据二分查找法中，计算mid的公式：$mid = low + (high-low)*（1/2）$，将其中的$1/2$替换成0.618即可。

使用斐波那契数列

斐波那契数列指的是这样一个数列：1、1、2、3、5、8、13、21、34、……
推导公式如下：
$F(1)=1，F(2)=1, F(n)=F(n - 1)+F(n - 2)（n ≥ 3，n ∈ N^*）$

斐波那契数列的特点是：随着n的增大，两个相邻项的比值会逐渐逼近0.618。

因此可以使用斐波那契数列来进行分割。

那么选择哪种方式好呢？
  • 使用乘法计算，涉及到浮点数的乘法运算，这相比加减法更耗时。
  • 使用斐波那契数列，需要预先存储含有斐波那契数列的数组，且有序数组的初始长度得是斐波那契数。因此，相比乘法计算更耗内存。

通常，更关注时间复杂度，因此常使用斐波那契数列，而这也是该查找算法的名称来源。

若有序数组的长度并非斐波那契数，还能使用斐波那契查找算法吗？

可以使用斐波那契查找算法。
只需在有序数组后面增加若干个最大值（即填充数组），直至新的有序数组的长度是斐波那契数为止。

当有序数组的长度为斐波那契数时，其后每次分割所得数组的长度必然也是斐波那契数。

对一有序数组按黄金比例分割，那是较长的部分在前还是较短的部分在前？

无所谓。

对有序数组按黄金比例分割只要求：分割后的两个部分，其比例接近黄金比例即可。

若有序数组经过填充，且查找到的数位于填充部分，怎么办？

因为填充时，填充的是有序数组（从小到大）的最大值，因此当发现查找到的数等于最大值时，返回最大值的位置下标即可。

如何判断停止？

若斐波那契数组的下标小于2，则就不用再分割了，因为斐波那契数组已经用完。

要进行斐波那契查找，则输入数组长度必须大于等于2，若输入数组长度小于2如何处理？

若有序数组是升序的，则在数组末尾填充最大值，直至数组长度等于2为止。

具体步骤？

  1. 根据有序数组长度计算斐波那契数组。
  2. 根据有序数组长度适当地进行填充。
  3. 对有序数组进行黄金分割。
  4. 判断关键字位于哪个部分？
  5. 再黄金分割，如此重复。

代码

复杂度

假设总共有n个元素。

时间复杂度

时间复杂度$logn$。

空间复杂度

斐波那契查找只需要为数组进行适当填充，其增加的空间和数组长度不成比例关系，因此空间复杂度为$O(1)$。

使用场景、条件或者限制

  1. 数组有序。
  2. 数组内元素不重复。

20200309-插值查找

插值查找

思想

思想来源

插值查找是对二分查找的改进，是基于如下问题“为什么每次都要对半，能不能四分之一或者其它比例呢？”而做出改进。

举个例子：

我们在一本英文字典里面查找apple这个单词的时候， 你肯定不会从字典中间开始查找， 而是从字典开头部分开始翻，因为觉得这样的找法才是比较快的。

根据上述例子，想到： 如果能在查找前较准确地预测关键字在数组中的位置，由此确定每次查找的比例，那这样设计出的查找方法能比二分查找提高更多的性能！ 这就是插值查找的思想。

如何根据数组和关键字来确定查找比例？

在二分查找中，确定比例的方式是：$mid = (low+high)/2$。
上式等价于：$mid = low + (high-low)*（1/2）$，其中$(1/2)$就是二分查找的查找比例。

假设关键字为key，有序数组为nums。

  1. 先判断key是否在nums[low]~nums[high]范围内，若是，则继续，反之说明当前范围内中不存在key，退出。
  2. $查找比例=(key - nums[low]) / (nums[high] - nums[low])$。
  3. 因此$mid = low + (high-low)*（(key - nums[low]) / (nums[high] - nums[low])）$。

假设总共有n个元素。

时间复杂度

不明白计算过程。

空间复杂度

不需要额外空间，所以空间复杂度为$O(1)$。

使用场景

  1. 在元素数值均匀分布的有序数组里面， 用这种方法查找是很快的。特别的，对绝对均匀分布的数组（相邻元素差值相同）， 插值查找用一次比较就能查找成功。

20200307-二分查找

二分查找

思想

二分查找（Binary Search）算法，也叫折半查找算法。二分查找的思想非常简单，有点类似分治的思想。二分查找针对的是一个有序的数据集合，每次都通过跟区间的中间元素对比，将待查找的区间缩小为之前的一半，直到找到要查找的元素，或者区间被缩小为 0。

若数列中没有想要的元素，该如何停止？

检测到最后时，下标low和high必然重叠，然后检查此时的数是否是要查找的数。
若是，则返回下标；若不是，则下标low将加一或者下标high将减一。
因此，可以通过判断high小于low，来说明数列中没有要查找的元素，那么就停止查找。

示例

以数组1, 2, 4, 5, 6, 7, 9, 12, 15, 19, 23, 26, 29, 34, 39，在数组中查找26为例。

代码

复杂度

假设总共有n个元素。

时间复杂度

每次查找的区间大小就是$n$，$n/2$，$n/4$，…，$n/2^k$，其中k就是循环的次数。
最坏的情况是找完整个数列都没有找到，即$n/2^k$取整后>=1，令$n/2^k$=1，
可得$k=log_2n$，所以最坏时间复杂度可以表示$O()=O(logn)$。

空间复杂度

不需要额外空间，所以空间复杂度为$O(1)$。

使用场景

  • 只适用于数组结构。
  • 数组必须有序。
  • 适用于大数据量。因为二分查找中还有计算mid的开销，若数据量过小，还是顺序查找快。

20200305-基数排序

基数排序

基数排序的发明可以追溯到1887年赫尔曼·何乐礼在打孔卡片制表机上的贡献。

思想

最低位优先法，简称LSD法

先从最低位开始排序，再对次低位排序，直到对最高位排序后得到一个有序序列。

示例

初始数组序列为：15,25,105,78,34,21,32,41。

  1. 按照个位数大小依次入桶。

    然后将桶中数依次倒出（对于同一个桶中的数按先进先出顺序倒出），结果为:21,41,32,34,15,25,105,78。

  2. 按十位数大小依次入桶。

    将桶中数依次倒出，结果为：105,15,21,25,32,34,41,78。

  3. 按百位上数大小依次入桶，没有百位的数则按百位为0入桶。

    将桶中数倒出，结果为：15,21,25,32,34,41,78,105。

最高位优先法，简称MSD法

先从最高位开始排序，再逐个对各分组按次高位进行子排序，循环直到最低位。

仍以序列：15,25,105,78,34,21,32,41为例，从最高位百位依次入桶，只有105有百位，其他百位按0算；检测每个桶中的数据。当桶中的元素个数多于1个的时候，要对这个桶递归进行下一位的分组。

20200304-桶排序

桶排序

思想

一般排序算法的时间复杂度是O(NlogN)，说明当数据量N越大，则运行所需时间越长。

那么考虑：能否将N个数据均匀地分成M个组，使每个组内的数据量约为N/M个；然后分别对每个组内的数据进行排序；最后合并每个组的数据，即完成对N个数据的排序。这就是桶排序的思想。

桶排序的步骤

  1. 设置固定数量的空桶。
  2. 使用的映射函数把数据放到对应的桶中。
  3. 对桶内数据进行排序。
  4. 拼接桶内数据。

实例

  1. 设置空桶数量为5个，由最大值110和最小值7确定每个桶的范围20.8=(110-7+1)/5 。
  2. 遍历原始数据，根据映射函数，将数据放到对应的桶中。例如：数字7，桶索引值为0，计算公式为floor((7 – 7) / 20.8)；数字36，桶索引值为1，计算公式floor((36 – 7) / 20.8)。
  3. 使用其它排序算法分别对各个桶内数据进行排序。
  4. 顺序合并各个桶内数据。

时间复杂度

时间复杂度分两部分考虑：

  1. 循环使用的映射函数把N个数据放到对应的桶中，其时间复杂度为O(N)。
  2. 使用其它排序算法对每个桶内的数据进行排序，其时间复杂度为 ∑ O(NilogNi) ，Ni 为第i个桶的数据量。

当N个数据均分到M个桶中时，得到平均时间复杂度：O(N)+O(M*(N/M)*log(N/M))。
当N个数据几乎都分到1个桶中时，得到最坏时间复杂度：O(N)+O(N*log(N))。
当桶的数量M等于数据数量N时，得到最佳时间复杂度：O(N)。

空间复杂度

桶排序中，需要创建M个桶的额外空间和N个数据的额外空间，所以桶排序的空间复杂度为 O(N+M)。
